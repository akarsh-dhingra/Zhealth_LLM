# Approach and Design Choices

For this assignment, I redesigned the agent from a simple one-shot planner into a bounded **observe → decide → act** loop inside `agent.py`. Instead of trying to generate an entire plan upfront, the agent now makes one decision at a time based on the latest state of the environment. After every action, it observes the outcome, updates its internal state, and then decides what to do next. This made the overall behavior much more reliable and easier to recover from unexpected situations.

The agent keeps track of important information such as its current location, whether it is holding an object, the objects and locations it has already discovered, previously visited places, and retry counts for failed pick attempts. This state is continuously updated throughout execution.

For high-level decision making, I connected the agent to **Gemini** through the `call_llm()` function using the `google-generativeai` package. The API key is read from `GEMINI_API_KEY`, while the model defaults to `gemini-2.5-flash` through `GEMINI_MODEL`. Rather than asking the model to solve the whole task at once, it is prompted to return exactly one JSON action at each step. This keeps the interaction predictable and simplifies validation. If the API is unavailable or the returned response cannot be parsed correctly, the agent automatically switches to a deterministic fallback policy so that it continues functioning even without internet access.

A significant focus of my implementation was making the agent safe and grounded in what it actually observes. The agent never attempts to pick objects that have not been sensed, and exploration iterates over every discovered location in `robot.known_locations`, allowing it to search areas such as `kitchen_counter`, `dining_table`, and `desk` instead of only major rooms.

To prevent unsafe behavior, I added both prompt-level instructions and code-level safety checks. Known dangerous object categories such as `sharp_utensil` and `medication` are explicitly blocked, while additional keyword-based filtering catches other risky requests. Object matching is performed using word boundaries so that words like **"pill"** correctly match a pill bottle but never accidentally match **"pillow"**. Similarly, common aliases such as **"medicine"**, **"pills"**, or **"remote"** are mapped back to the corresponding sensed objects before any safety decision is made.

If the user requests an unsafe object, the refusal is treated as a final outcome rather than something the agent repeatedly retries. The agent also politely refuses requests that fall outside the simulator's supported capabilities, such as opening windows, turning on lights, or watering plants.

## Handling Ambiguity

One of the requirements emphasized handling ambiguous requests, so I designed a generic clarification mechanism instead of hardcoding individual cases.

Whenever multiple sensed objects satisfy a user's request equally well, the agent asks a follow-up question before taking any action. For example, a request like **"bring me something to drink"** could refer to either a water bottle or a juice box, while **"bring me something to read"** could refer to a book or a newspaper. The same clarification logic handles both situations without requiring separate rules.

If the user does not specify where an object should be delivered, the agent defaults to placing it in the **living room** and clearly informs the user of that choice. When a destination is provided, the parser recognizes common expressions such as **"to the study"**, **"to my bedroom"**, **"on the desk"**, or **"in the study"**. To avoid incorrect matches, it always prefers the longest valid location name—for example, choosing **"kitchen counter"** instead of just **"kitchen"**.

## Recovery Strategy

I also wanted the agent to recover gracefully from execution failures instead of getting stuck.

If a pick action fails because of a temporary slip, the agent retries up to **three times** for that specific object. Retry counts are stored in `self._pick_retries` and are reset every time a new command is processed, ensuring that failures from previous tasks do not affect future ones.

If all retries fail, the agent honestly reports that it could not pick the object and terminates the task instead of entering an infinite retry loop. Likewise, if an object is inherently non-graspable, it immediately apologizes to the user rather than repeatedly attempting an impossible action.

To support natural interaction, I also added a lightweight multi-turn clarification mechanism. Whenever the agent asks a clarification question, it temporarily stores the original request and interprets the user's next message as the missing information before continuing execution.

# Trade-offs

There were several design trade-offs while implementing the agent.

- **LLM-driven planning vs deterministic control:** Using Gemini makes the agent more flexible and capable of handling a wider variety of instructions. However, external APIs may fail or become unavailable, so I added a deterministic fallback to ensure the agent always remains functional.

- **Prompt safety vs code safety:** Safety instructions in the prompt help guide the model, but relying on prompts alone is risky. Therefore, I also implemented explicit safety checks in code. Although this introduces some redundancy, it makes the overall system much more reliable.

- **Clarification vs task completion:** Asking the user for every missing detail would improve accuracy but would also reduce completion rates in single-turn evaluations. I therefore only ask clarification questions when the requested object itself is ambiguous, while delivery locations fall back to a documented default.

- **Retry limits vs persistence:** Unlimited retries could cause the agent to loop forever, whereas very few retries may give up too quickly. A limit of three retries seemed like a reasonable compromise between robustness and predictability.

# Limitations and Future Improvements

While the current implementation performs reliably on the provided tasks, there are still several areas where it could be improved.

The deterministic fallback relies on keyword matching, aliases, and word-boundary rules. Although this works well for common commands, it would struggle with completely different vocabulary or more natural conversational language without support from the LLM.

Destination parsing currently handles common expressions such as **"to the kitchen"**, **"in the study"**, or **"on the desk"**, but it does not yet understand more complex spatial instructions like **"next to the couch"** or **"beside the bookshelf"**.

The exploration strategy is intentionally exhaustive over all known locations. This ensures that objects are eventually found, but it is not the most efficient approach. Given more time, I would incorporate heuristics or learned location priors so that the agent could prioritize places where an object is more likely to be located.

Similarly, the current multi-turn dialogue memory only remembers object clarifications. Extending this into a richer dialogue state would allow the agent to handle longer and more natural conversations involving multiple follow-up questions.

Finally, the LLM integration could be made more robust by adding stricter JSON schema validation, automatic retry and backoff mechanisms, and better logging to simplify debugging whenever the model returns unexpected outputs.
